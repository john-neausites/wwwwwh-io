# GitHub Copilot Instructions for wwwwwh.io

## Project Overview
wwwwwh.io is a P2P/S2S digital file sharing platform with physical key authentication, designed to provide users with perpetual access to their digital content through decentralized architecture and community-driven sustainability.

## Core Objective: Perpetual Content Access
The fundamental mission is to ensure users can access their content forever through:
- **Decentralized Storage**: P2P network with no single point of failure
- **Community Seeding**: Incentivized peer storage for content persistence
- **Hardware Security**: Physical key authentication for tamper-proof access
- **Legal Compliance**: Public domain and first-sale doctrine content distribution
- **Zero Vendor Lock-in**: Open protocols and user-controlled data

## Architecture Principles

### 1. User Data Sovereignty
- Users control their own content and encryption keys
- Platform never stores user content, only coordinates peer connections
- Hardware keys provide cryptographic proof of ownership
- Content can be migrated between platforms using open standards

### 2. Perpetual Access Mechanisms
- **IPFS Content Addressing**: Immutable content hashes for permanent identification
- **Community Redundancy**: Multiple peers store popular content
- **Seeding Incentives**: Economic rewards for long-term content hosting
- **Graceful Degradation**: Platform can operate with minimal infrastructure
- **Export Capabilities**: Users can extract their content and metadata at any time

### 3. Privacy and Security
- **End-to-End Encryption**: All content encrypted before leaving user devices
- **Zero-Knowledge Architecture**: Platform cannot decrypt or access user content
- **Minimal Metadata**: Only essential coordination data stored on servers
- **Hardware Authentication**: FIDO2 keys prevent account takeover
- **Audit Trails**: Cryptographic logs for transparency without privacy invasion

## Technology Stack Rationale

### Node.js Prototype → Rust Production
- **Rapid Development**: TypeScript for quick iteration and testing
- **Production Performance**: Rust for memory safety and concurrent P2P handling
- **Gradual Migration**: Service-by-service replacement strategy
- **Ecosystem Compatibility**: Node.js for broader library ecosystem during development

### PostgreSQL with ltree
- **Hierarchical Data**: Efficient tree operations for content categorization
- **Activity Mapping**: Complex queries for contextual content discovery
- **Metadata Storage**: Structured data for peer coordination
- **ACID Compliance**: Reliable state management for P2P network

### libp2p + WebRTC
- **Standards-Based**: Open protocols prevent vendor lock-in
- **NAT Traversal**: Direct peer connections through firewalls
- **Protocol Flexibility**: Upgrade transport layers without breaking compatibility
- **Browser Integration**: WebRTC enables web-based P2P without plugins

### Perpetual Access Implementation Patterns

#### 1. Content Addressing
Always use content-addressable storage:
```typescript
// Good: Content identified by cryptographic hash
const contentId = await ipfs.add(fileBuffer);
const record = {
  hash: contentId.path, // Immutable identifier
  userKey: hardwareKeyId, // Owner verification
  metadata: encryptedMetadata // Privacy-preserving info
};
```

#### 2. Redundancy Management
Implement community storage incentives:
```typescript
// Track and reward content seeding
const seedingReward = calculateReward({
  contentRarity: getRarityScore(contentHash),
  storageContribution: peer.bandwidthProvided,
  communityValue: content.downloadCount
});
```

#### 3. Export/Import Functions
Always provide data portability:
```typescript
// Export user's complete content catalog
const exportData = {
  content: user.ownedContent.map(item => ({
    hash: item.hash,
    decryptionKey: item.encryptedKey,
    metadata: item.metadata
  })),
  peers: user.trustedPeers,
  activityHistory: user.contextualPreferences
};
```

### Naming Conventions
- **Activities**: Use CORE_ACTIVITIES keys (e.g., `cook`, `workout`, `relax`)
- **Content Paths**: ltree format (e.g., `audio.music.albums.classical`)
- **API Endpoints**: RESTful with context (e.g., `/api/v1/content/recommend`)
- **Database Tables**: Snake_case (e.g., `content_items`, `peer_connections`)

### Error Handling
Always preserve user access in failure scenarios:
```typescript
// Graceful degradation pattern
try {
  return await p2pNetwork.getContent(hash);
} catch (networkError) {
  // Fall back to local cache
  const cached = await localStorage.getContent(hash);
  if (cached) return cached;
  
  // Fall back to backup peers
  return await backupPeers.requestContent(hash);
}
```

## Security Considerations

### Hardware Key Integration
- Never store private keys on servers
- Use WebAuthn for browser-based authentication
- Implement key rotation without losing content access
- Support multiple keys per user for redundancy

### Content Verification
- Verify content integrity using cryptographic hashes
- Implement reputation system for peer trust
- Detect and prevent content poisoning attacks
- Use digital signatures for content authenticity

### Privacy Protection
- Implement differential privacy for usage analytics
- Use onion routing for metadata protection
- Minimize data collection to essential coordination only
- Provide user controls for all data sharing
